System Architecture

Full document: Section 1 - Executive Summary

The connector runs natively inside Hummingbot as a Python ExchangePyBase subclass. No Gateway, no Docker, no middleware. Writes go through Stellar RPC. Reads go through Horizon.

System Architecture
flowchart TD TRADER["Trader"] subgraph HB["Hummingbot Process - Python"] STRAT["Strategy Layer\npure_market_making / avellaneda_market_making"] EXCH["StellarExchange\nExchangePyBase subclass"] OBT["OrderBookTracker"] UST["UserStreamTracker"] CHP["ChannelPool"] end subgraph NET["Network"] RPC["Stellar RPC\nJSON-RPC 2.0"] HOR["Horizon API\nREST and SSE"] end TRADER --> STRAT STRAT --> EXCH EXCH --> OBT EXCH --> UST EXCH --> CHP CHP -- "sendTransaction\ngetTransaction\ngetNetwork" --> RPC OBT -- "GET + SSE /order_book" --> HOR UST -- "SSE /accounts/id/transactions" --> HOR EXCH -- "GET /accounts/id\nGET /fee_stats" --> HOR RPC -- "PENDING / SUCCESS / FAILED" --> EXCH HOR -- "orderbook, transactions, balances" --> EXCH classDef hbot fill:#F97316,stroke:#C2410C,color:#fff classDef stellar fill:#0096D6,stroke:#0074A8,color:#fff classDef horizon fill:#1E3A5F,stroke:#3B82F6,color:#fff classDef user fill:#7C3AED,stroke:#5B21B6,color:#fff class STRAT,EXCH,OBT,UST,CHP hbot class RPC stellar class HOR horizon class TRADER user

Why RPC for writes, Horizon for reads? The RFP notes the connector should ideally use Stellar RPC. We use RPC for all write operations (sendTransaction, getTransaction, getNetwork). For orderbook reads, Stellar RPC has no endpoint for OfferEntry records - the persistent ledger state that makes up the SDEX orderbook. Horizon is the only production-ready API for this data. The market data layer is abstracted behind a single interface so it can migrate to Portfolio APIs or any future RPC orderbook endpoint without touching the rest of the connector.

Transaction Submission Flow

Every order placement and cancellation follows this flow. The channel account is the transaction source (pays the fee, owns the sequence number). The master account is the operation source (owns the offers).

Channel Account Architecture

Full document: Section 5 - Channel Account Architecture (5.1-5.4)

Stellar allows one transaction per account per ledger (~5-7 seconds). A market maker refreshing a three-level spread needs 12 simultaneous transactions. Channel accounts solve this.

Channel Account Lifecycle
flowchart TD A([Uninitialized]) --> B([Ready]) B --> C([Acquiring]) C -- "semaphore acquired" --> D([InUse]) C -- "30s timeout exceeded" --> E([TimedOut]) E --> F([ChannelUnavailableError]) D --> G([Building]) G -- "TransactionBuilder.build()" --> H([Signing]) H -- "sign both keypairs" --> I([Submitted]) I -- "release_channel()" --> B I -- "TX_BAD_SEQ" --> J([SequenceError]) J -- "invalidate, re-fetch" --> G I -- "TX_INSUFF_FEE" --> K([FeeError]) K -- "re-fetch /fee_stats, rebuild" --> G classDef normal fill:#0096D6,stroke:#0074A8,color:#fff classDef ok fill:#16A34A,stroke:#15803D,color:#fff classDef err fill:#DC2626,stroke:#B91C1C,color:#fff classDef warn fill:#D97706,stroke:#B45309,color:#fff classDef start fill:#1E3A5F,stroke:#3B82F6,color:#fff class A,C,D,G,H,I start class B,J,K normal class F err
  • Channels are funded at 2.0 XLM, replenished when they drop below 1.5 XLM
  • All replenishments go into a single multi-operation transaction (one sequence number, one fee)
  • If the main account's available XLM is insufficient, the provisioner logs a warning and skips the cycle
  • Sequence numbers are tracked locally - no network round-trip before each submission
  • On sequence error, the local value is invalidated and re-fetched automatically

Two-Phase Order ID and Offer Lifecycle

Full document: Section 6 - Order Placement (6.1-6.4)

Stellar generates offer IDs after a transaction confirms on the ledger. We use the transaction hash as a provisional ID immediately, then swap it for the real offer ID once the transaction lands.

Two-Phase Order ID Resolution

Buy vs Sell Balance Locking

SELL order - ManageSellOffer(selling=XLM):
  Required:  available_XLM >= order_amount

BUY order - ManageBuyOffer(selling=USDC to buy XLM):
  Required:  available_USDC >= (order_amount x price)

XDR Parsing Details

  • TransactionMeta versions 0-4 are all handled for protocol compatibility
  • Created/Updated entries are LedgerEntry objects (access via .data.type)
  • Removed entries are LedgerKey objects (access directly via .type - no .data wrapper)
  • OfferEntry.amount is always in the selling asset. For buy orders, divide by price to get base amount
# CREATED or UPDATED - LedgerEntry
if change.created.data.type == LedgerEntryType.OFFER:
    offer = change.created.data.offer    # OfferEntry

# REMOVED - LedgerKey (no .data wrapper)
if change.removed.type == LedgerEntryType.OFFER:
    offer_id = str(change.removed.offer.offer_id)

Fill Amount Calculation

SELL order (selling=XLM):
  remaining_base = offer_entry.amount / 10_000_000

BUY order (selling=USDC):
  remaining_quote = offer_entry.amount / 10_000_000
  remaining_base  = remaining_quote / order_price

Orderbook Data

Full document: Section 4.2 - Horizon, The Read Path

The SDEX orderbook is made up of OfferEntry records sitting in the ledger as persistent state. There is no event stream that can reconstruct it. Horizon's /order_book endpoint is the only public API that reads it directly.

Each SSE event from Horizon's /order_book stream delivers a complete orderbook snapshot for both bid and ask sides - not an incremental diff. On reconnection, the data source performs a full GET request to re-establish baseline state before resuming the SSE stream.

EndpointMethodPurpose
/order_bookGET + SSEOrderbook snapshot and per-ledger streaming
/accounts/id/transactionsSSEFill detection via account transaction stream
/accounts/idGETAccount balances and subentry count
/accounts/id/offersGETOpen offer count for reserve calculation (batched)
/fee_statsGETClassic transaction fee estimation (p90 percentile)

Balance Management

Full document: Section 8 - Balance Management 
available_XLM = total_XLM
              - 1.0 XLM                       # base account reserve
              - 0.5 XLM x num_trustlines       # per trustline subentry
              - 0.5 XLM x num_open_offers       # per open offer subentry
              - xlm.selling_liabilities          # XLM committed to open sell orders

available_USDC = total_USDC - usdc.selling_liabilities

All values from Horizon are strings. Always use Decimal(balance["balance"]) - never float(). Status polling uses a single batched call to /accounts/id/offers for all open orders rather than one request per order.

Every error code comes from stellar_sdk.xdr.TransactionResultCode and ManageSellOfferResultCode - typed Python enums. No string comparisons. Sequence and fee errors are retried automatically. Balance and unknown errors are surfaced to the strategy.
