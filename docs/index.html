Executive Summary

This connector integrates the Stellar DEX with Hummingbot as a direct Python ExchangePyBase subclass -the same pattern the XRPL connector uses. It runs inside the Hummingbot process. There is no external server, no extra Docker container, and no middleware. That setup is required for the connector to be officially merged into the Hummingbot repository and maintained alongside everything else in the codebase.

The table below covers the five main architectural calls and the reasoning behind each one.

Decision
Rationale
Direct Python connector
Hummingbot requires an ExchangePyBase subclass for official inclusion. The Stellar Python SDK covers everything -no JavaScript runtime needed.
Stellar RPC for writes
sendTransaction and getTransaction are the correct interfaces for transaction submission and confirmation. This is what they are built for.
Horizon SSE for reads
The SDEX orderbook is persistent OfferEntry ledger state. No event stream can reconstruct it. Horizon's /order_book endpoint is the only public API that reads it directly.
Channel accounts
Stellar enforces one transaction per account per ledger. A market maker refreshing a spread needs many simultaneous transactions -channel accounts solve this at the protocol level.
Two-phase offer ID
Stellar generates offer IDs at the ledger level after a transaction confirms. The transaction hash serves as a temporary ID until the real one is available.

Technology Stack

The dependency surface is intentionally small. The stellar-sdk handles XDR encoding and transaction construction. All network calls use aiohttp directly with persistent sessions -not the SDK's built-in HTTP client. This avoids async compatibility issues between the SDK and Hummingbot's event loop, and gives us full control over connection pooling.

LayerTechnologyWhy
Trading frameworkPython 3.10+, HummingbotStrategy execution and connector interface
Network I/Oaiohttp 3.xAll HTTP and SSE connections. Persistent sessions -no new TCP handshake per request
Transaction constructionstellar-sdk 10.0.0+XDR encoding, keypair management, operation building only. Not used for networking.
Decimal arithmeticPython decimal.DecimalEvery monetary value. Never float -floating-point rounding errors in financial math compound over time.
Async timeoutasync_timeout packageasyncio.timeout was added in Python 3.11. Hummingbot targets 3.10+, so we use the backport.

stellar-sdk version matters. TransactionMetaV4 -what Protocol 23 mainnet produces -requires stellar-sdk 10.0.0 or higher. Earlier versions silently fail to parse transaction metadata. The connector checks the SDK version at startup and raises a clear error if it is too old.

Transport Layer

The connector uses two separate transport mechanisms -Stellar RPC for writing, Horizon for reading. This is not a workaround. It reflects what each API is actually built to do.

4.1 Stellar RPC -The Write Path

Every transaction goes through Stellar's JSON-RPC 2.0 interface. When we submit a transaction, we get a PENDING status back and then poll getTransaction until it confirms or fails. The polling interval is three seconds with a maximum of 210 seconds -just beyond the 180-second transaction timeout, so we never stop watching before the transaction can possibly expire.

Transaction Submission Flow

4.2 Horizon -The Read Path

The SDEX orderbook is made up of OfferEntry records sitting in the ledger as persistent state. An offer placed six months ago is still a valid open order today. There is no event that fires when it was created -it just exists. That means you cannot reconstruct the current orderbook from a transaction or event feed. You have to read it directly from the ledger, and Horizon's /order_book endpoint is the only public API that does this.

Why not Stellar RPC for the orderbook? Stellar RPC is purpose-built for Soroban smart contract interactions. It has no endpoint for reading classic OfferEntry records from the BucketList. When Portfolio APIs add orderbook support, the connector's market data layer is abstracted behind a single interface that makes the switch a one-file change.

Each SSE event from Horizon's /order_book stream delivers a complete orderbook snapshot for both bid and ask sides -not an incremental diff. On reconnection, the data source performs a full GET request to re-establish baseline state before resuming the SSE stream. There is no incremental delta to reconcile.

Channel Account Architecture

Stellar allows exactly one transaction per account per ledger. A market maker refreshing a three-level spread needs to cancel six orders and place six new ones simultaneously. From a single account, those twelve transactions would take roughly sixty seconds. With a channel pool, they happen in one ledger close.

5.1 How It Works

Each channel account has its own independent sequence number and pays its own transaction fees. The operations inside each transaction specify the main trading account as their source -so all offers are created and cancelled under the main account regardless of which channel submitted the transaction. The main account holds all assets. A channel account holds only enough XLM for its base reserve and fees.

5.3 Sequence Number Tracking

We track sequence numbers locally to avoid fetching them from the network before every transaction. Once a channel loads its sequence from Horizon for the first time, we advance the local counter ourselves each time a transaction is submitted. On any sequence error, we invalidate the local value and re-fetch on the next use.

Horizon returns sequence as a string. The response is "sequence": "100" -not an integer. It must be converted with int(response["sequence"]) before being passed to stellar_sdk.Account. Passing a string causes a TypeError when TransactionBuilder tries to compute sequence + 1.

# First use -fetch from Horizon and convert to int
response = await horizon_get(f"/accounts/{channel_pubkey}")
local_sequence = int(response["sequence"])   # String to int -required

# Build with local sequence
account_obj = stellar_sdk.Account(channel_pubkey, local_sequence)
tx = TransactionBuilder(source_account=account_obj, ...).build()

# TransactionBuilder.build() increments account_obj.sequence in-place
# Sync our local tracker to this new value
local_sequence = account_obj.sequence

# On TX_BAD_SEQ
def invalidate_sequence(self):
    self.local_sequence = None   # Re-fetch on next use

5.4 Fee Provisioner

Channel accounts slowly drain XLM from transaction fees. A background task checks balances every thirty seconds and replenishes anything below 1.5 XLM. Channels are replenished to 2.0 XLM. If the main account's available XLM is insufficient to cover all pending replenishments, the provisioner logs a warning and skips the cycle until the user adds funds. All replenishments go into a single multi-operation transaction from the main account -one sequence number, one fee, everything topped up in a single ledger close.

6.2 Two-Phase Order ID

Stellar generates offer IDs after a transaction confirms on the ledger. Hummingbot's _place_order needs an exchange_order_id to return synchronously. We use the transaction hash as a temporary ID immediately -it's deterministic and computable before broadcast -then swap it for the real offer ID once the transaction lands.

6.3 Balance Check -BUY vs SELL

Sell orders and buy orders lock different assets. Getting this wrong means the strategy thinks it has more available balance than it does.

SELL order -ManageSellOffer(selling=XLM):
  Required:  available_XLM >= order_amount
  On confirmation:  XLM selling_liabilities increases

BUY order -ManageBuyOffer(selling=USDC to buy XLM):
  Required:  available_USDC >= (order_amount x price)
  On confirmation:  USDC selling_liabilities increases

6.4 Immediate Fill Handling

When a limit order crosses existing offers and fills completely at submission, the result contains MANAGE_OFFER_DELETED. No offer was created, so there is no offer ID. The connector skips the OPEN state and moves the order directly to FILLED.

7.2 LedgerEntryChanges Parsing

Two things to get right here. First, LEDGER_ENTRY_STATE is always a before-state snapshot that immediately precedes a LEDGER_ENTRY_UPDATED record. Skip STATE and process only UPDATED. Second, created and updated entries are LedgerEntry objects that expose type via .data.type. Removed entries are LedgerKey objects with no .data wrapper -type is accessed directly via .type.

# CREATED or UPDATED -LedgerEntry
if change.created.data.type == LedgerEntryType.OFFER:
    offer = change.created.data.offer    # OfferEntry

# REMOVED -LedgerKey (no .data wrapper)
if change.removed.type == LedgerEntryType.OFFER:
    offer_id = str(change.removed.offer.offer_id)

7.3 Fill Amount Calculation

OfferEntry.amount is always in the selling asset, stored in units of 10^-7. For sell orders this is the base asset directly. For buy orders this is the quote asset, which needs to be divided by the order price to get the base amount. This distinction shows up in three places: the user stream fill calculation, status polling from /accounts/id/offers, and _all_trade_updates_for_order.

SELL order (selling=XLM):
  remaining_base = offer_entry.amount / 10_000_000

BUY order (selling=USDC):
  remaining_quote = offer_entry.amount / 10_000_000
  remaining_base  = remaining_quote / order_price

Balance Management

Stellar locks XLM for every subentry on an account. Getting the reserve calculation wrong means the bot tries to trade XLM it does not actually have available, which produces failed transactions.

available_XLM = total_XLM
              - 1.0 XLM                       # base account reserve
              - 0.5 XLM x num_trustlines       # per trustline subentry
              - 0.5 XLM x num_open_offers       # per open offer subentry
              - xlm.selling_liabilities          # XLM committed to open sell orders

available_USDC = total_USDC - usdc.selling_liabilities

The open offer count requires a separate call to /accounts/id/offers -the subentry_count field on the account response does not break down by type.

Status polling uses a single batched call to /accounts/id/offers for all open orders rather than one request per order. This keeps Horizon usage flat regardless of how many pairs are running.

All values from Horizon are strings. Always use Decimal(balance["balance"]) -never float(). Floating-point rounding in financial math compounds over thousands of transactions.

selling_liabilities must be subtracted per asset. USDC locked in open buy orders shows up in the USDC balance's selling_liabilities, not XLM's.

Phase 2 Scope

The core SDEX connector does not include AMM integration. This section describes the Phase 2 extension adding classic liquidity pool support and Soroban AMM routing. It is documented here so Phase 2 can proceed without rearchitecting anything.

Every Soroban contract interaction requires three steps. You cannot skip the simulation -it is how the protocol determines what ledger keys the contract will read and write, which must be declared in advance.

The connector inherits Hummingbot's native security model. We do not add separate credential management on top of it.

Private key storageHummingbot encrypts all credentials with the user's password. Keys for the main account and channel accounts are stored in Hummingbot's encrypted config files and never leave the machine.
Channel account isolationEach channel holds only enough XLM for base reserve and fees. All trading assets stay in the main account. A compromised channel key cannot access trading funds.
Network passphrase bindingAll transactions include the network passphrase in their hash computation. A transaction signed for Testnet is cryptographically invalid on Mainnet.
Startup validationBoth the RPC and Horizon endpoints are checked against the configured passphrase before any operations proceed. Mismatches are caught before a trade is placed.

Key Design Decisions

Why a direct Python connector and not a Gateway?

Hummingbot accepts ExchangePyBase subclasses as first-class connectors that go into the main repository. Gateway connectors are used when an exchange requires a JavaScript SDK that cannot run in Python. The Stellar Python SDK covers everything we need. A Gateway would mean a Node.js server, a separate Docker container, SSL certificates, and an HTTP hop between Hummingbot and the connector -none of that is necessary here.

Why Horizon for orderbook data and not Stellar RPC?

Stellar RPC is designed for Soroban smart contract interactions. It has no endpoint for reading OfferEntry records, which are classic ledger entries in the BucketList. The SDEX orderbook cannot be built from event streaming because open offers have no historical event -they are persistent state. Horizon's /order_book endpoint reads that state directly and is the only production-ready way to do it today. When Portfolio APIs add orderbook access, the market data layer is behind a single abstraction that makes the swap a one-file change.

Why local sequence number tracking?

Fetching the channel's current sequence from Horizon before every transaction adds 100 to 300 milliseconds per submission. With five channels running during a fast order refresh cycle, that latency adds up. Local tracking eliminates the cost entirely. On any sequence error, we invalidate the local value and re-fetch -recovery is automatic with no manual intervention.

Why batch all replenishments into one transaction?

Replenishing five channels with five separate Payment transactions would require five sequential submissions from the main account -five sequence numbers, five fees, five ledger closes. Batching them into one multi-operation transaction uses one sequence number and one fee. Stellar allows up to 100 operations per transaction, so this scales without any changes to the architecture.
